Understanding DNS Basics

What is DNS?

DNS (Domain Name System) is like the phone book of the internet. It translates human-readable website names (like google.com) into computer-readable IP addresses (like 172.217.164.110).

Simple Analogy: Phone Book

Just like you look up "John Smith" in a phone book to find his phone number, your computer looks up "google.com" in DNS to find Google's server address.

How DNS Works:

DNS Resolution Process 1. User types "example.com" in browser ‚Üì 2. Browser asks DNS resolver: "What's the IP for example.com?" ‚Üì 3. DNS resolver checks cache, then queries DNS servers ‚Üì 4. DNS server responds: "example.com = 93.184.216.34" ‚Üì 5. Browser connects to 93.184.216.34 ‚Üì 6. Website loads!

üåê Domain Name

Human-friendly name like "amazon.com"

üî¢ IP Address

Computer address like "54.239.28.85"

‚ö° DNS Resolver

Service that looks up IP addresses

üìö DNS Records

Database entries mapping names to addresses

Understanding Content Delivery

What is Content Delivery?

Content delivery is about getting your website's files (images, videos, web pages) to users as quickly as possible, no matter where they are in the world.

Simple Analogy: Pizza Delivery

Instead of having one pizza shop serve the entire city (slow delivery to far areas), you have multiple pizza shops in different neighborhoods. Users get pizza from the closest shop = faster delivery!

The Problem Without CDN:

Without CDN - Single Server User in Tokyo -------- 8000 miles -------- Server in New York (Slow!) (Your website) User in London ------- 3500 miles -------- Server in New York (Slow!) (Your website) User in New York ----- 10 miles ---------- Server in New York (Fast!) (Your website)

The Solution With CDN:

With CDN - Multiple Edge Locations User in Tokyo -------- 50 miles --------- CDN Edge (Tokyo) (Fast!) ‚Üì Your Content User in London ------- 30 miles --------- CDN Edge (London) (Fast!) ‚Üì Your Content User in New York ----- 10 miles --------- CDN Edge (New York) (Fast!) ‚Üì Your Content

Amazon Route 53

What is Amazon Route 53?

Route 53 is AWS's DNS service that translates domain names to IP addresses and routes users to your applications. Think of it as the internet's GPS system.

Why "Route 53"?

Named after Route 53, a famous highway, and port 53, which is the standard port for DNS traffic.

Key Features:

  • Domain Registration: Buy and manage domain names
  • DNS Hosting: Host DNS records for your domains
  • Health Checks: Monitor application health
  • Traffic Routing: Route users based on location, health, etc.
  • Global Network: Fast DNS resolution worldwide

Why Use Route 53:

  • Highly available and reliable DNS service
  • Integrates seamlessly with other AWS services
  • Advanced routing policies for better performance
  • Built-in health monitoring
  • Cost-effective with pay-as-you-go pricing

Route 53: Lab 1 - Create Hosted Zone

üß™ Lab: Set Up DNS for Your Domain

Goal: Create a hosted zone and configure basic DNS records

Prerequisites:

  • AWS account with Route 53 access
  • A domain name (you can use a test domain or register one)

Step-by-Step Instructions:

  1. Navigate to Route 53
    • Go to AWS Console ‚Üí Services ‚Üí Route 53
    • Click "Hosted zones" in the left menu
  2. Create Hosted Zone
    • Click "Create hosted zone"
    • Domain name: Enter your domain (e.g., "example.com")
    • Type: Public hosted zone
    • Click "Create hosted zone"
  3. Note the Name Servers
    • Route 53 creates NS (Name Server) records
    • Copy the 4 name servers (e.g., ns-123.awsdns-12.com)
    • You'll need these for your domain registrar
  4. Create A Record
    • Click "Create record"
    • Record name: Leave blank (for root domain)
    • Record type: A
    • Value: Enter an IP address (e.g., 93.184.216.34)
    • TTL: 300 seconds
    • Click "Create records"
  5. Create CNAME Record
    • Click "Create record"
    • Record name: "www"
    • Record type: CNAME
    • Value: Your root domain (e.g., "example.com")
    • Click "Create records"

Expected Output:

  • Hosted zone created with NS and SOA records
  • A record pointing your domain to an IP address
  • CNAME record for www subdomain

Route 53: Lab 2 - Health Check & Failover

üß™ Lab: Set Up Health Check and Failover Routing

Goal: Create health checks and configure failover between two endpoints

Step-by-Step Instructions:

  1. Create Health Check for Primary Server
    • Route 53 Console ‚Üí Health checks ‚Üí Create health check
    • Name: "Primary-Server-Health"
    • What to monitor: Endpoint
    • Protocol: HTTP
    • IP address: Your primary server IP
    • Port: 80
    • Path: / (or /health)
    • Create health check
  2. Create Health Check for Secondary Server
    • Repeat above steps for secondary server
    • Name: "Secondary-Server-Health"
    • Use different IP address
  3. Create Primary Record with Failover
    • Go to your hosted zone
    • Create record ‚Üí Record name: "app"
    • Record type: A
    • Routing policy: Failover
    • Failover record type: Primary
    • Value: Primary server IP
    • Health check: Select "Primary-Server-Health"
    • Record ID: "primary"
  4. Create Secondary Record with Failover
    • Create record ‚Üí Record name: "app" (same as primary)
    • Record type: A
    • Routing policy: Failover
    • Failover record type: Secondary
    • Value: Secondary server IP
    • Health check: Select "Secondary-Server-Health"
    • Record ID: "secondary"
  5. Test Failover
    • Use nslookup or dig to query "app.yourdomain.com"
    • Should return primary server IP when healthy
    • Simulate primary server failure
    • Wait 2-3 minutes, query again
    • Should now return secondary server IP

Expected Output:

  • Health checks monitoring both servers
  • Automatic failover when primary server fails
  • DNS queries return appropriate server IP based on health

Amazon CloudFront

What is Amazon CloudFront?

CloudFront is AWS's Content Delivery Network (CDN) that delivers your website content to users from the closest geographic location, making your site faster worldwide.

Simple Analogy: Global Library System

Instead of having one library in New York that everyone must visit, you have copies of popular books in libraries worldwide. People get books from their local library = faster access!

Key Features:

  • Global Edge Locations: 400+ locations worldwide
  • Caching: Stores copies of your content closer to users
  • SSL/TLS: Built-in security and encryption
  • Origin Shield: Additional caching layer
  • Real-time Metrics: Monitor performance and usage

Why Use CloudFront:

  • Faster website loading times globally
  • Reduced server load and bandwidth costs
  • Built-in DDoS protection
  • Easy integration with other AWS services
  • Pay only for data transfer you use

Goal: Create a CloudFront distribution to serve content from an S3 bucket

Prerequisites:

  1. Prepare S3 Bucket

    • This allows CloudFront to access your bucket
  6. Test Your Distribution

    • Open it in browser

Amazon API Gateway

What is Amazon API Gateway?

API Gateway is like a smart receptionist for your applications. It receives requests from clients (mobile apps, websites), routes them to the right backend services, and sends responses back.

Simple Analogy: Hotel Receptionist

When you call a hotel, the receptionist (API Gateway) takes your request, connects you to the right department (room service, housekeeping, etc.), and ensures you get the right response.

Key Features:

  • Request Routing: Direct requests to appropriate backend services
  • Authentication: Control who can access your APIs
  • Rate Limiting: Prevent abuse with request throttling
  • Monitoring: Track API usage and performance
  • Caching: Cache responses for better performance

Why Use API Gateway:

  • Centralized entry point for all API requests
  • Built-in security and authentication
  • Automatic scaling to handle any traffic volume
  • Easy integration with AWS Lambda and other services
  • Pay only for API calls you receive

Networking & Delivery Best Practices

üèÜ Golden Rules for AWS Networking

üåê Route 53 Best Practices

  • Use health checks for high availability
  • Implement geolocation routing for global apps
  • Set appropriate TTL values (300-3600 seconds)
  • Use alias records for AWS resources
  • Monitor DNS query patterns

‚ö° CloudFront Best Practices

  • Use Origin Access Control (OAC) for S3
  • Set appropriate cache behaviors and TTLs
  • Enable compression for text content
  • Use custom error pages
  • Monitor cache hit ratios

üö™ API Gateway Best Practices

  • Always use authentication (API keys, IAM, etc.)
  • Implement rate limiting and throttling
  • Use caching for frequently requested data
  • Version your APIs properly
  • Monitor API usage and errors

üí∞ Cost Optimization

  • Use CloudFront to reduce data transfer costs
  • Choose appropriate CloudFront price classes
  • Cache API responses when possible
  • Monitor usage with CloudWatch
  • Use reserved capacity for predictable workloads

‚ö†Ô∏è Common Mistakes to Avoid:

  • Not using HTTPS/SSL certificates
  • Setting cache TTLs too high or too low
  • Forgetting to invalidate CloudFront cache after updates
  • Not implementing proper API authentication
  • Ignoring monitoring and logging
